Stufe 1 · Ausgangspunkt

Ein Wunsch ist noch keine belastbare Anforderung

Wir starten bewusst mit einem Satz, der verständlich klingt, aber viele Fragen offenlässt.

ROH

„Das System soll Dokumente hochladen können."

Schon erkennbar

  • Es gibt ein System.
  • Dokumente sollen übertragen werden.
  • Eine Upload-Funktion wird erwartet.

Noch unklar

  • Wer lädt was hoch?
  • Welche Formate und Größen?
  • Wie schnell und wie sicher?
  • Was gilt als erfolgreich?
Stufe 2 · Nutzung verstehen

Stakeholder und Nutzungskontext

Bevor wir die Funktion präzisieren, klären wir, wer sie in welcher Situation benötigt.

Wer?

Angemeldete Projektmitarbeitende mit Upload-Berechtigung.

Wozu?

Projektdokumente revisionssicher im zugeordneten Projektraum bereitstellen.

Wann?

Während der laufenden Projektarbeit über freigegebene Browser.

Kontextsatz

„Eine berechtigte Projektmitarbeiterin lädt ein Projektdokument in den zugeordneten Projektraum hoch, damit das Team mit der aktuellen Fassung arbeiten kann."

Stufe 3 · Funktion klären

Funktionale Anforderung

Die Funktion beschreibt das beobachtbare Systemverhalten – noch ohne technische Lösung vorzugeben.

Stufe 4 · Produktqualität

ISO 25010 als Qualitätsbrille

Die Norm liefert keine fertigen Anforderungen. Sie hilft, systematisch die richtigen Qualitätsfragen zu stellen.

1. Funktionale Eignung

Frage: Unterstützt die Funktion den vollständigen fachlichen Ablauf?
NFA: PDF-, DOCX- und XLSX-Dateien müssen akzeptiert werden.

2. Leistungseffizienz

Frage: Wie schnell muss der Upload reagieren?
NFA: Dateien bis 20 MB müssen unter Referenzbedingungen innerhalb von 5 Sekunden verarbeitet werden.

3. Kompatibilität

Frage: In welchen Umgebungen muss die Funktion nutzbar sein?
NFA: Der Upload muss in den freigegebenen Versionen von Chrome, Edge und Safari funktionieren.

4. Interaktionsfähigkeit

Frage: Wie verständlich ist die Bedienung?
NFA: Ablehnungsgründe müssen in verständlicher Sprache angezeigt werden.

5. Zuverlässigkeit

Frage: Was passiert bei Abbruch oder Fehlern?
NFA: Fehlgeschlagene Uploads dürfen keine unvollständigen Dateien erzeugen.

6. Sicherheit

Frage: Wie werden Zugriff und Daten geschützt?
NFA: Nur berechtigte Personen dürfen Dateien in das jeweilige Projekt übertragen.

7. Wartbarkeit

Frage: Wie leicht lassen sich Regeln ändern?
NFA: Dateiformate und Größenlimits müssen konfigurierbar sein.

8. Flexibilität

Frage: Wie gut passt sich die Lösung neuen Rahmenbedingungen an?
NFA: Weitere Speicherziele müssen über eine Schnittstelle ergänzt werden können.

9. Safety

Frage: Welche schädlichen Auswirkungen müssen verhindert werden?
NFA: Eine neue Datei darf eine bestehende Fassung nicht unbeabsichtigt überschreiben.

Stufe 5 · Informationssicherheit

ISO 27001: vom Risiko zur Sicherheitsanforderung

Die Sicherheitsbetrachtung folgt einer nachvollziehbaren Kette: Informationswert, Bedrohung, Risiko, Maßnahme und Nachweis.

Informationswert

Vertrauliche Projektdokumente.

→
Bedrohung

Unbefugter Zugriff oder manipulierte Datei.

→
Risiko

Offenlegung, Veränderung oder Schadcode.

→
Maßnahme

Autorisierung, Prüfung, Protokollierung.

SEC-01 · Autorisierung

Das System muss vor jedem Upload prüfen, ob die angemeldete Person für das Zielprojekt berechtigt ist.

SEC-02 · Integrität

Das System muss die vollständig übertragene Datei anhand eines Hashwerts überprüfen.

SEC-03 · Schadsoftware

Die Datei muss vor der endgültigen Bereitstellung auf Schadsoftware geprüft werden.

SEC-04 · Audit

Jeder Upload muss mit Nutzerkennung, Projekt, Zeitstempel und Ergebnis protokolliert werden.

Stufe 6 · Konsolidierung

Die präzisierte Gesamtanforderung

Funktionale, qualitative und sicherheitsbezogene Aussagen bilden nun ein belastbares Requirements-Paket.

FA-01

Das System muss berechtigten Projektmitarbeitenden ermöglichen, PDF-, DOCX- und XLSX-Dateien bis 20 MB einem aktiven Projekt zuzuordnen, zu übertragen und nach erfolgreicher Prüfung im Projektraum bereitzustellen.

NFA-PERF-01

Unter Referenzbedingungen muss die Verarbeitung einer Datei bis 20 MB innerhalb von 5 Sekunden abgeschlossen sein.

NFA-USE-01

Bei Ablehnung muss das System den konkreten Grund und eine mögliche Korrektur verständlich anzeigen.

SEC-01

Vor der Annahme muss das System Projektberechtigung, Dateityp und Dateigröße serverseitig prüfen.

REL-01

Fehlgeschlagene Uploads dürfen keine unvollständigen oder sichtbaren Dateien erzeugen.

Akzeptanzkriterien mit Gherkin

Szenario: Berechtigte Person lädt eine gültige Datei hoch

Und sie besitzt Upload-Rechte für Projekt A

Und im Projektraum bereitgestellt
Und der Upload wird mit Nutzerkennung und Zeitstempel protokolliert
Szenario: Datei überschreitet die zulässige Größe

Angenommen eine Datei ist größer als 20 MB
Wenn die Projektmitarbeiterin den Upload startet
Dann wird die Datei nicht gespeichert
Und das System nennt die Größenbegrenzung
Und es entsteht keine unvollständige Datei

const maxBytes = 20 * 1024 * 1024;
const allowed = ["pdf", "docx", "xlsx"];

function validate(file) {
  const ext = file.name.split(".").pop().toLowerCase();
  if (!allowed.includes(ext)) return "Dateiformat nicht zulässig.";
  if (file.size > maxBytes) return "Datei ist größer als 20 MB.";
  return "";
}
ALLOWED = {"pdf", "docx", "xlsx"}
MAX_BYTES = 20 * 1024 * 1024

def upload(file, user, project):
    require_project_permission(user, project)
    validate_extension(file, ALLOWED)
    validate_size(file, MAX_BYTES)
    temp_path = save_to_quarantine(file)
    scan_for_malware(temp_path)
    final_path = commit_file(temp_path, project)
    write_audit_log(user, project, final_path)
    return {"status": "uploaded"}

HTML stellt Auswahl und Aktion bereit. JavaScript prüft vorab; Python nimmt die Datei an und verarbeitet sie.

Interaktionsfähigkeit

Beschriftung, Statusbereich und verständliche Fehlermeldungen machen die Funktion wahrnehmbar und bedienbar.

Performance Efficiency

Die 20-MB-Grenze wird im Browser und nochmals verbindlich auf dem Server geprüft.

ISO 27001 · Sicherheit

Berechtigung, zulässige Dateitypen, Quarantäne und Schadsoftwareprüfung schützen Zugriff, Integrität und Verarbeitung.

Reliability

Die Datei wird zunächst temporär gespeichert und erst nach erfolgreicher Prüfung endgültig übernommen.

Nachvollziehbarkeit

Das Audit-Log dokumentiert, wer welche Datei welchem Projekt hinzugefügt hat.
